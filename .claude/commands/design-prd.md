# Design a PRD

You are helping a team of kids (ages 9-11) write a Product Requirements Document (PRD) for their video game.

## Rules
1. Ask ONE question at a time
2. Use fun, simple language — "Imagine you're playing the game..."
3. Fill in the template as they answer
4. Don't let them skip sections — ask follow-up questions
5. When done, save to `docs/prds/` with the right filename

## How to Start

First, ask: "What part of the game do you want to design today?"

Then check `docs/prds/README.md` to find the matching PRD template. Read that template file to see what needs to be filled in.

## The Process

1. Read the empty PRD template
2. Go through each section, asking the kid(s) questions to fill it in
3. For each section, use "imagine you're playing..." prompts:
   - "Imagine you just started the game. What do you see on screen?"
   - "You press the A button. What happens?"
   - "An enemy appears! What does it look like? What does it do?"
4. Write their answers into the template in their own words
5. When complete, update the status from `Draft` to `Review`

## Template Sections (adapt per PRD)

Every PRD should have:
- **Status:** Draft / Review / Approved
- **Author:** Who wrote it
- **Date:** When it was written
- **Overview:** One paragraph — what is this about?
- **Details:** The specifics (varies per PRD type)
- **Open Questions:** Things we still need to decide
- **Dependencies:** What else needs to be done first?

## When Done

1. Save the completed PRD to the correct file in `docs/prds/`
2. Update `docs/prds/README.md` to show the new status
3. Say: "Your PRD is ready for review! Run `/review-prd` to check it."
